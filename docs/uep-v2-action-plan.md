## 验证材料策略（双层夹具，全程有效）

- **单元层**：微型合成夹具（快、可入库）；
- **集成层**：**真实数据集切片**（各 ≥100 条，HF 下载）——试金石与往返测试以真实切片为准；
- 切片获取存**脚本 + 校验和**，数据本体默认不入库（许可不明确的绝不入库）。
- 教训来源：ref/ 的"数据集"全是手造微型样本（ARC 仅 1 条），对玩具全绿证明不了通用性。

## 风险登记簿

| 风险 | 预案 |
|---|---|
| 首选 Runner 导出格式对不上假设 | 阶段 0 裁决②前做半天 spike 实证格式，再定稿 |
| 模型访问/费用超预期 | 最小切片 + 最便宜模型；预算红线由 0.2④ 裁定，触线即停并上报 |
| HF 真实切片下载受阻 | 镜像源备选；脚本可重放；本地缓存 |
| 外部评审人/零背景者缺位 | 降级路径：你本人验收为底线，外部人并入接触点阶段补位 |
| 时间盒超支 | 强制砍范围保节奏，被砍项显式挂起，不隐性顺延 |
| 会话中断/上下文丢失 | 文档+记忆+git 三重持久化（已实证有效）；每工作单元即提交 |

## 横切纪律（全程有效）

1. 每个工作单元以"测试绿 + git 提交"收尾，仓库永远可复现；**分支模型：trunk-based，直接 master 小步提交**；
2. 用户闸口（SPEC 签字、每原型评审卡、命名与原型边界裁决）必须等人，其余全速；
3. 讨论一次一个问题；文档状态诚实（DRAFT 就标 DRAFT）；
4. 未跑通不称"完成"；
5. **计划变更规则**：本计划的任何变更经用户确认，并在文首 changelog 记一行；
6. **旧项目零参考政策（2026-07-03 用户裁定，取代先前"摘片段"政策）**：完全不参考 ref/ 的代码与数据，只承接其需求理念——clean-room 重写；夹具与样例一律新建或取自公开源；
7. **阶段收口仪式**：每阶段结束做一次复盘（"是否还能完善"）+ git tag，再进下一阶段。
