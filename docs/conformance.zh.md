# UEP 一致性工具包（conformance kit）

[English version](conformance.en.md)

给**新建数据集的创建者**：三层自查，一条命令。通过 = 你的数据集能被
标准工具链机械消费——校验、组卷、导出、判分意图自明。

```bash
uep conform my_ds/items.jsonl
```

## 三层检查

| 层 | 检查什么 | 不过意味着什么 |
|---|---------|---------------|
| 1 schema 合法 | 每行是合法 EvalItem（坏行报行号，双语报错） | 数据还不是 UEP |
| 2 试金石可消费 | 每条可被**对应原型的标准消费程序**渲染（消费程序源码零数据集名） | 打分意图不自明——标准 Runner 无法机械判分 |
| 3 清单一致 | manifest 的 size / task_types / languages 与条目**机械复核** | 清单漂移（清单说一套、数据是另一套） |

## 边界与约定

- **custom 原型**（受控逃生舱，SPEC §3.6）跳过第 2 层并计数提示——逃生舱
  本就不在试金石辖区；
- **manifest.json 缺失** = 第 3 层跳过并提示（items.jsonl 单独分发是合法形态）；
- **仅用 llm_judge 判分的条目**：schema 合法但不过第 2 层——这是有意的：
  L1 一致性 = 可机械消费；主观判分超出 L1 承诺范围；
- 退出码：`0` 通过 / `1` 未通过 / `2` 用法错误；`--lang zh|en` 切换输出语言。

## 与验证金字塔的关系

`uep conform` 是本仓库 L1 机械层对第三方数据集的打包：同一批试金石、
同一套清单复核逻辑，我们自己的 12 个接入数据集每次 CI 都在过（`make demo`
关卡 1 可复现）。你的数据集过了 conform，就站在与它们相同的地基上。
