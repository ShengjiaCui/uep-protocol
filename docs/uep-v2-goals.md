# UEP v2 目标文档

> **地位**：本文档是 UEP v2 的**目标权威**。需求文档（`.claude/PRPs/prds/uep-v2.prd.md`）与后续 SPEC 从本文档推导，冲突时以本文档为准。
> **形成方式**：2026-07-03 与项目负责人逐问对齐（一次一问、逐层确认），所有引号内文字为负责人原话。
> **状态**：已确认（负责人逐条确认：目标结构 "Good"、范围三含义 "I Agree"、其余各条当轮确认）。

---

## 1. 使命（综合表述）

让评测数据在任何领域、任何评测工具、中英两种语言之间**无损流动**——UEP 成为评测数据的通用协议，最终成为业界创建新 benchmark 时的参考标准。

> 用户 JTBD 原话："作为一个 AI 从业人员，我可以通过 UEP，把所有的评测集有效转化后，便利的使用。"

## 2. 目标结构（四层）

| 层 | 内容（原话为准） | 性质 |
|---|---|---|
| **终局真北** | "我们的协议，被业界接纳，大家再新建 benchmark，会参考我们的协议" | 成功的定义 |
| **一年后** | "我们的协议，能够在各个领域都优雅兼容使用" | 中期图景 |
| **资格线** | 能力四关卡（见 §6）——必要条件，**非成功本身** | 质量下限 |
| **失败红线** | "没人使用" | 失败的定义 |

**三条推论（已确认）**：
1. 真北是**采纳**，不是能力——能力只保证"配得上被参考"；
2. **新 benchmark 创建者**是一等用户——"被参考"发生在创建时刻，UEP 原生创作必须比自造 JSONL 更省事；
3. **冷启动是中心风险**——构建期就要有外部接触点，不能闭门造完再发布。

## 3. 覆盖范围：Agent 任务空间原则

> 原话："凡是 AI agent 要覆盖的领域，都在我们的承载之内，当然，执行可以有先后顺序，但初心不变。"

三条设计含义（已确认 "I Agree"）：

1. **协议表达力的上限按 Agent 任务空间设定**——环境、多轮轨迹、工具调用、状态变化、复合验证，从第一天就在协议骨架里，不做"以后再补"；
2. **静态 QA 是 Agent 任务的退化形态**——设计自上而下：按最复杂情形定骨架，简单情形是简化投影；
3. **协议必须自带演进能力**——Agent 领域是开放集合，"都在承载之内"只能靠扩展机制兑现，且演进治理中含人工仲裁点。

## 4. "优雅兼容"的定义

**六判据**（原话以反例给出："不易用、不易懂、不兼容多语言（至少英文、中文），不是人类常用，违法行业习惯，不符合专业领域通用认知与专业知识"，翻正如下）：

| # | 判据 | 含义 |
|---|---|---|
| 1 | 易用 | 上手与日常操作低摩擦 |
| 2 | 易懂 | 读得懂，不需要读源码才能理解 |
| 3 | 多语言兼容 | 中英起步（详见 §5） |
| 4 | 贴合人类常用 | 用熟悉的惯例，不发明生僻构造 |
| 5 | 遵循行业习惯 | 不违背评测/数据工程的通行做法 |
| 6 | 尊重领域专业认知 | 承载某领域数据时，该领域专家看着眼熟 |

**判据冲突的裁决原则**（原话）："优先专业，难以抉择时，可以引入 human 决定。"
——跨域统一与领域行话冲突时，**领域专业优先**；协议治理设人工仲裁点，不搞纯机械规则。

**既有反例基线**（原话）："我们的协议在不同的数据集类似数据类型之间是没有字段复用，这就是瞎搞。"
——同类语义必须复用同一规范字段，这是优雅的底线（三条硬性设计原则之一，详见 PRD）。

## 5. 双语平权承诺

> 目标句（原话）："US 用户和中文用户，都能无损优雅，完整使用。" 展开受托于 Claude（2026-07-03），如下四层：

1. **数据内容层（硬约束）**：中/英/混语内容无假设承载；条目与数据集携带 BCP-47 语言元数据；Unicode 规范化策略显式（NFC），禁止默认 ASCII 化；
2. **打分语义层（硬约束）**：Verifier 不得内嵌英文假设——大小写折叠、空白、全半角、中文标点等参数化且对中文安全；**"无损"包括打分行为对中文内容同样正确**；
3. **协议词汇层**：字段名/原型名/枚举值保持英文（循判据 5"行业习惯"）；规范附官方中英术语对照表；
4. **工具与文档层**：协议规范、README、教程中英双语同步；CLI 输出与报错支持 zh/en；双语文本冲突以英文版为准（可复议）。

## 6. 资格线：能力四关卡

通过全部四关 = 协议**配得上被参考**；不构成成功（成功 = 采纳）。

| # | 关卡 | 通过标准 |
|---|---|---|
| 1 | 通用消费者试金石 | 存在不含任何数据集名字的通用消费程序，对所有已接入数据集全部工作 |
| 2 | 无损往返 | 每个双向格式 X→UEP→X 语义等价，自动化测试覆盖 |
| 3 | 五分钟上手 | 零背景用户从 CSV 到 ≥2 个真实 Runner 跑通，≤5 分钟、零代码 |
| 4 | SPEC 零漂移 | 每条 FR 绑定验收测试，CI 绿 = 承诺兑现 |

## 7. 采纳愿景（背景，外围已停车）

> 原话三级飞轮："1）我们会推出一个基于 UEP 的数据集开放平台，我们把常用数据都'翻译过来'方便大家用。2）我们自己会建专业的数据集，基于该协议给客户使用。3）客户或用户，用我们的协议，创建数据集，发布到我们的平台来使用。"

第一个外部使用者画像：来平台取用"已翻译好的常用数据集"的从业者——先享受翻译成果，后被协议吸附。

> **范围裁定**（原话）："我们先只探讨 UEP 协议，外围暂时不讨论。以后再扩展。"
> ——平台与商业化在**停车场**（非放弃）；当前一切讨论与建设只围绕协议本体。

## 8. 边界（非目标）

- **不做 Runner / 评测执行引擎**——中立性根基（继承自旧项目，未被推翻）；
- **本阶段不讨论**：数据集开放平台、专业数据集商业形态（停车场，见 §7）；
- **不与通用数据集标准竞争**（Croissant/HF Datasets），定位评测垂直层（待竞品补充扫描复核）。

## 9. 停车场（已挂起、待以后处理）

| 事项 | 挂起原因 |

| 专业数据集的客户画像与商业形态 | 同上 |
| 任务原型清单 v1、还原策略等 SPEC 细节 | 目标先行，经用户同意后解冻下钻 |
